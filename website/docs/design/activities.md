# ActivityGroups & Activities

<Lead>An activity is one retryable action — publish an event, persist a record, update a downstream balance. An activity group coordinates a cohesive set of them for a single business concern. Activities are shared across markets and stay thin, delegating protocol details to clients.</Lead>

Groups whose behaviour varies by market carry their dimensions in the second column; the rest are generic and run the same way everywhere.

| Activity / ActivityGroup | Generic / Dimensions | State transition | What it does |
| --- | --- | --- | --- |
| `PersistPendingPaymentActivity` | Generic | input → `PENDING` | • Inserts the payment into the `idempotency_checker`, `trans_dtl` (transaction detail), and `trans_lfcyc_event` (transaction lifecycle event) tables.<br />• If a row is already present the request is a duplicate; otherwise it is a genuine first-time (idempotent) request.<br />• On success, publishes a `PENDING` lifecycle event.<br />• Invoked from `InitiatedToPendingStage`. |
| `IdempotencyCheckActivity` | Generic | — | • Attempts to insert a row into the `idempotency_checker` table for the API being called.<br />• If the row already exists the request is a duplicate; if not, this is the first time the request has been seen.<br />• Invoked from any workflow that needs an idempotency check. |
| `PaymentValidationActivityGroup` | `accountType`, `requiresArPosting`, `requiresRealtimeClearing`, `requiresMandateAuthorization` | `PENDING` → `ACCEPTED` / `SCHEDULED` / `DECLINED` | • Makes external calls where needed and evaluates the data to decide whether the payment is valid.<br />• A valid immediate payment becomes `ACCEPTED`; a valid future-dated payment becomes `SCHEDULED`; an invalid one becomes `DECLINED`.<br />• Invoked from the `CreateImmediatePaymentWF` and `CreateSchedulePaymentWF` workflows. |
| `PaymentValidationOnExecutionActivityGroup` | `accountType`, `requiresArPosting`, `requiresRealtimeClearing`, `requiresMandateAuthorization` | `SCHEDULED` / `ALLOCATED` → `ACCEPTED` / `DECLINED` | • Runs when a scheduled payment is due to execute.<br />• Calls the systems needed to gather information that may have changed since scheduling — some calls can only be made after data has been fetched from another system first.<br />• Checks whether the scheduled payment is still valid: still-valid becomes `ACCEPTED`, otherwise `DECLINED`.<br />• Invoked from `ExecuteScheduledPaymentWF`. |
| `PaymentStateTransitionActivity` | Generic | Various (payment level) | • Given a payment id with its current and previous states, updates the `trans_dtl` table to the new status.<br />• Adds a new row to the `trans_lfcyc_event` table for that state.<br />• Publishes the matching lifecycle event.<br />• Invoked inside every stage that moves a payment between states at the payment level. |
| `PaymentSplitStateTransitionActivity` | Generic | Various (split level) | • The split-level counterpart: updates the `split_trans_dtl` table to the new status for a split-payment id.<br />• Adds a row to the `split_trans_lfcyc_event` table.<br />• Publishes the split-level lifecycle event.<br />• Invoked inside every stage that handles split payments. |
| `PaymentScheduledNotificationActivityGroup` | `accountType`, `requiresArPosting` | — | • Notifies the relevant downstream systems once a payment has been scheduled.<br />• Invoked from `PendingToScheduledStage`. |
| `PaymentDeclinedNotificationActivityGroup` | `accountType`, `requiresArPosting` | — | • Notifies the relevant downstream systems when a payment is declined — whether it was declined during an immediate payment or while being scheduled.<br />• Invoked from `PendingToDeclinedStage`. |
| `PaymentAllocatingActivityGroup` | Generic | `PENDING` / `SCHEDULED` → `ALLOCATING` | • Requests the allocations for a corporate payment.<br />• For a corporate immediate payment, the payment reaches `ACCEPTED` (by way of `ALLOCATING` and `ALLOCATED`) only once its allocations are fetched; if they cannot be fetched, it is `DECLINED`.<br />• For a corporate scheduled payment, it moves to `ALLOCATING` on the scheduled date, or a set number of days before it.<br />• Invoked from `ToAllocatingStage`. |
| `PaymentAllocatedActivityGroup` | Generic | `ALLOCATING` → `ALLOCATED` | • Receives the allocations back from GPA (Get Corporate Payment Allocations).<br />• The validation Billpay performs on the received allocations is still to be defined.<br />• Invoked from `AllocatingToAllocatedStage`. |
| `PaymentExecutionActivityGroup` | `accountType`, `requiresArPosting`, `requiresRealtimeClearing` | `ACCEPTED` → `PROCESSING` | • Runs three actions in parallel, recording each in the notification tracker table: sends the payment to the clearing system, decreases the Accounts Receivable balance, and increases Open-To-Buy in Authorization.<br />• These steps can vary depending on the payment's dimensions.<br />• Invoked from `AcceptedToProcessingStage`. |
| `PaymentFulfillmentActivityGroup` | `accountType` | `PROCESSING` → `PROCESSED` | • In parallel, notifies Accounting and Balance & Control, recording each in the notification tracker table.<br />• Then notifies Communications, also recorded in the notification tracker table.<br />• Invoked from `ProcessingToProcessedStage`. |
| `PaymentSplitsCreationActivity` | Generic | `ACCEPTED` (full) → `ACCEPTED` (split) | • For each split, creates a row in the split transaction-detail and `split_lfcyc_event` tables in the `ACCEPTED` state.<br />• Publishes the split-level lifecycle event for that transition.<br />• Invoked from the `CreateImmediatePaymentWF` and `ExecuteScheduledPaymentWF` workflows. |
| `PaymentCancelValidationActivityGroup` | `accountType` | — | • Checks the payment's current state in the database.<br />• If it is eligible, makes the external calls needed to confirm the cancellation is allowed, returning true when it is.<br />• Invoked from `CancelPaymentWF`. |
| `PaymentCancellationActivityGroup` | `accountType`, `requiresArPosting` | `SCHEDULED` / `ACCEPTED` → `CANCELLED` | • Notifies the various downstream systems once a payment has been cancelled.<br />• Invoked from `CancelPaymentWF`. |
| `PaymentReturnValidationActivity` | Generic | — | • Checks the returned payment's current state in the database, continuing only if it is valid.<br />• Invoked from `ProcessReturnedPaymentWF`. |
| `PaymentReturnExecutionActivityGroup` | `accountType` | `PROCESSING` / `PROCESSED` / `PAID` → `RETURNED` | • Notifies the various downstream systems once a payment has been marked `RETURNED` in the database.<br />• Invoked from `ToReturnedStage`. |
| `PaymentRepresentmentEligibilityActivityGroup` | `accountType`, `requiresArPosting`, `requiresRealtimeClearing`, `requiresMandateAuthorization` | — | • Makes the external calls needed to decide whether a returned payment can be re-presented (retried for settlement).<br />• Returns true if it is eligible, false otherwise.<br />• Invoked from `ProcessReturnedPaymentWF`. |
| `PaymentRepresentmentCreationActivityGroup` | Generic | `RETURNED` (presentment seq 1) → `REPRESENTING` (presentment seq 2) | • Creates a new row in the `trans_dtl` and `trans_lfcyc_event` tables in the `REPRESENTING` status and publishes a lifecycle event.<br />• Takes the return from presentment sequence 1 (`RETURNED`) to sequence 2 (`REPRESENTING`).<br />• Invoked from `ReturnedToRepresentingStage`. |
| `PaymentRepresentmentValidationActivityGroup` | `accountType`, `requiresArPosting`, `requiresRealtimeClearing`, `requiresMandateAuthorization` | — | • Makes the external calls needed to confirm, on the day the representment is due to run, that it is still valid to execute.<br />• Returns true if so, false otherwise.<br />• Invoked from `ProcessRepresentmentWF`. |
| `PaymentRepresentmentExecutionActivityGroup` | `accountType`, `requiresArPosting`, `requiresRealtimeClearing`, `requiresMandateAuthorization` | `REPRESENTING` → `REPRESENTED` | • Sends the payment to the clearing system.<br />• Then notifies the various downstream systems in parallel once it has been re-presented (`REPRESENTED`).<br />• Invoked from `RepresentingToRepresentedStage`. |
| `MapNewPaymentIdToPreviousIdActivity` | Generic | — | • Writes a row into the `ORIG_TRANS_REFER_MAP` table linking the old and new payment ids.<br />• Preserves the audit trail when a scheduled payment is replaced by an updated one.<br />• Invoked from `UpdatePaymentWF`. |

For the layering that governs what may call what, see the [component model](./principles.md).
